# 코드를 읽지 않는 시대, 엔지니어는 무엇을 읽어야 하는가

> **Source**: [flowkater.io](https://flowkater.io/posts/2026-02-19-code-reading-era/)
> **Author**: Tony Cho
> **Archive Date**: 2026-02-25

AI가 코드 생성을 대신하는 시대에 엔지니어가 집중해야 할 것은 개별 코드 라인이 아니라, 시스템 설계·아키텍처·컨텍스트라는 주장을 연구와 사례 기반으로 정리한 글.

---

## 핵심 연구 결과

### Anthropic 연구
- AI 보조를 사용한 개발자들이 이해도 테스트에서 **17% 낮은** 점수 기록
- 단, 동일 도구를 사용해도 **접근 방식에 따라** 결과가 크게 달랐음
  - **최악**: 코드를 전적으로 AI에 위임한 사람
  - **최고**: AI에 개념적 질문을 하고 코드는 직접 작성한 사람

### Berkeley 연구
- 비개발자가 AI로 작성한 코드를 엔지니어가 리뷰·수정하는 데 직접 작성하는 것보다 **더 많은 시간** 소요
- 역량 격차는 사라지지 않고 **형태만 바뀜**

## AI 시대 코드 리딩의 변화

### Harness Engineering (Ben Shoemaker)
라인별 코드 리뷰 대신 **사양, 자동 테스트, 린터, 보안 스캔**으로 코드를 검증. 코드 자체보다 코드를 둘러싼 인프라에 집중.

### 코드의 상품화 (Evan Armstrong)
코드 생성은 누구나 할 수 있게 되지만, 프로덕션 코드를 **거버닝**하는 것 — 무엇이 존재하고 누가 변경할 수 있는지 이해하는 **컨텍스트 레이어** — 은 여전히 전문 영역.

### Steve Yegge의 AI 채택 8단계
Level 1(AI 거부) → Level 8(다수 에이전트 오케스트레이션). "누군가 수동으로 코드 리뷰하는 것을 보면 안타깝다"

## Finish Line vs. Compounding Game (Kent Beck)

| | Finish Line Game | Compounding Game |
|---|---|---|
| 정의 | X 목표 달성 후 종료 | 각 기능이 미래 기능의 토대 |
| AI 적합성 | 잘함 | 보장 못함 |
| 핵심 | 단일 작업 완료 | 확장성·유지보수성 |

> "AGENTS.md 파일만으로는 복리 게임을 이길 수 없다"

소프트웨어 개발 대부분은 Compounding Game. AI는 Finish Line에 강하지만 장기적 아키텍처 일관성은 보장하지 못한다.

## AI는 거울이다 (Jeremy Utley)

> "게으른 사람은 더 게을러지고, 날카로운 사람은 더 날카로워진다"

### 고성과 AI 사용 패턴 3가지
1. **Generation then Understanding** — 코드 생성 후 구현에 대해 후속 질문
2. **Hybrid Code-Explanation** — 코드 생성과 개념 토론 혼합
3. **Conceptual Exploration** — AI에 개념 질문, 코드는 직접 작성

최저 성과자: AI에 전적으로 위임한 사람

## 주의해야 할 함정

### Context Ceiling
모르는 것을 모르면 AI에게 올바른 질문을 할 수 없다. 보안 전문성이 없으면 보안 리뷰를 요청하지 않는다.

### Dracula Effect
AI와의 집중 협업은 인지적으로 매우 소모적.
> "2~3개 프로젝트를 동시에 진행하면 한두 시간만 일해도 하루치 정신 에너지가 거의 다 소진됩니다"

### Children of Magenta
항공 자동화 사례 — 자동조종에 전적으로 의존한 조종사는 수동 개입 판단력을 상실. 엔지니어도 코드 읽기 역량을 유지해야 한다.

## 엔지니어가 길러야 할 것

1. **스펙과 컨텍스트 정의** — AGENTS.md, 도메인 용어 사전, 제약 조건 정의, 테스트 프레임워크 설계
2. **대화, 질문이 아닌** — "AI에게 네 컨텍스트를 완전히 이해할 때까지 질문하라고 시켜라" (Utley)
3. **비판적 코드 읽기** — 모든 테스트가 통과해도 프로덕션이 깨질 때, 코드 읽기는 대체 불가
4. **미래 지향적 설계** — 현재 요구사항뿐 아니라 확장 가능성(futures)에 투자

## 변하지 않는 것

- 시스템 정확성에 대한 **비판적 사고**
- 확장성을 보장하는 **아키텍처 비전**
- 맥락적 결정을 가능케 하는 **도메인 이해**
- 작동하는 코드와 좋은 코드를 구별하는 **품질 판단**
